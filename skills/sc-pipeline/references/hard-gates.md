# 硬闸门机制

## 什么是硬闸门

硬闸门（Hard Gate）是发布前必须通过的**自动化强制质量检查**。与 Checkpoint（用户确认点）不同，闸门由系统自动执行检查，不通过则阻塞流程，无法进入发布阶段。

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  mining  │ ──→ │ writing  │ ──→ │ visuals  │ ──→ │  review  │ ──→ │publishing│
│          │     │ /imaging │     │          │     │ (硬闸门)  │     │          │
│ 用户确认  │     │ 用户确认  │     │ 用户确认  │     │ 自动检查  │     │ 用户确认  │
└──────────┘     └──────────┘     └──────────┘     └────┬─────┘     └──────────┘
                                                        │
                                              ┌─────────┴─────────┐
                                              │  不通过 → 回退修复  │
                                              │  通过 → 进入发布    │
                                              └───────────────────┘
```

## 闸门检查项

review 阶段按顺序执行以下检查项。根据平台不同，检查项略有差异：

| 检查项 | xhs | wechat | 说明 | 对应 Skill |
|--------|:---:|:------:|------|-----------|
| 排版格式化 | — | ✅ | Markdown 转公众号友好排版 | sc-format-markdown |
| 内容审核 | ✅ | ✅ | 合规+事实+链接+平台红线 | sc-content-review |
| 图片压缩 | ✅ | ✅ | 所有图片压缩为 WebP | sc-compress-image |

### 1. 内容审核（sc-content-review）— 两个平台都执行

**检查维度**：
- **合规性**：敏感话题、违规内容、平台红线
- **事实准确性**：数据、引用、论断是否有据可查
- **链接有效性**：文中链接是否可访问（如有）
- **平台规则**：
  - xhs：诱导关注/分享话术、标签合规、医疗/金融等特殊领域内容
  - wechat：政治敏感、诱导分享、虚假宣传、外链/二维码合规、原创声明

**结果分级**：

| 级别 | 处理方式 | 示例 |
|------|---------|------|
| `critical` | **阻塞发布**，必须修复后重新审核 | 敏感词、虚假宣传、事实错误 |
| `warning` | **需用户确认**，用户决定是否修复 | 错别字、表述可优化、标签不精准 |
| `pass` | 直接通过 | 无问题 |

### 2. 图片压缩（sc-compress-image）— 两个平台都执行

**要求**：
- 所有图片必须压缩为 WebP 格式
- 单张图片大小控制在合理范围（封面 ≤ 2MB，正文 ≤ 1MB）
- 压缩后图片清晰度可接受
- 压缩失败的图片需重新生成或替换

**WeChat 额外要求**：
- 图片宽度 ≤ 1080px
- 封面图比例 2.35:1

### 3. 排版格式化（sc-format-markdown）— 仅 WeChat

**要求**：
- Markdown 转换为公众号友好的排版格式
- 配图自动插入到正确位置
- 图注完整
- 标题层级、段落间距、引用样式等排版规范
- 格式化过程中的 warnings 需处理

## 闸门不通过时的处理流程

```
闸门检查执行
      │
      ▼
┌─────────────┐
│ 结果判定     │
└──────┬──────┘
       │
  ┌────┴────┐
  │         │
 通过     不通过
  │         │
  ▼         ▼
进入     ┌──────────────┐
发布     │ 展示问题清单   │
阶段     │ 标注 critical/ │
         │ warning       │
         └──────┬───────┘
                │
          ┌─────┴─────┐
          │           │
     critical      warning
          │           │
          ▼           ▼
   ┌────────────┐  ┌────────────┐
   │ 阻塞流程    │  │ 询问用户    │
   │ 引导修复    │  │ 继续/修改   │
   └─────┬──────┘  └─────┬──────┘
         │               │
         ▼               ▼
   用户修复后      ┌────────┴────────┐
   重新执行审核    │                 │
   （从第一步      用户选择         用户选择
    开始）        "继续发布"       "先修改"
                  │                 │
                  ▼                 ▼
             闸门放行          修复后重新审核
             进入发布
```

### 具体步骤

1. **展示问题清单**：将所有问题以列表形式展示给用户，标注 critical/warning 级别和具体位置
2. **critical 问题处理**：
   - 明确告知"以下问题必须修复后才能继续"
   - 引导用户修改对应内容（文案回到 writing 阶段，图片回到 visuals/imaging 阶段）
   - 修改完成后，**从 review 阶段第一步重新执行**（排版→审核→压缩）
3. **warning 问题处理**：
   - 告知"以下问题建议优化，但不强制"
   - 询问用户选择："继续发布"或"先修改"
   - 选择"继续发布"→ 闸门放行，进入 publishing
   - 选择"先修改"→ 同 critical 处理流程
4. **图片压缩失败处理**：
   - 标记失败的图片
   - 引导用户：重新生成该图片 / 替换为其他图片 / 调整压缩参数
   - 修复后重新压缩
5. **排版问题处理（WeChat）**：
   - 展示 format_warnings
   - 引导调整或手动修正 formatted.md
   - 修正后重新审核

## 闸门与 Checkpoint 的区别

| 维度 | 硬闸门（Hard Gate） | Checkpoint（用户确认点） |
|------|-------------------|----------------------|
| **执行方式** | 系统自动执行检查 | 人工（用户）确认 |
| **目的** | 质量底线保障，拦截严重问题 | 确保方向正确，用户满意 |
| **通过条件** | 所有 automated checks 通过 | 用户明确表示"OK/继续" |
| **不通过后果** | 阻塞流程，无法继续 | 回到当前阶段重做 |
| **出现位置** | 仅 review 阶段 | 每个阶段结束后都有 |
| **问题来源** | 规则引擎/AI审核/工具执行 | 用户主观判断 |
| **可跳过** | 不可跳过，critical 必须修复 | 用户可选择"跳过"（skipped 状态） |

> **简单理解**：Checkpoint 问用户"你满意吗？"，硬闸门问系统"这东西能发吗？"。两者互补，缺一不可。

## 闸门通过标准

闸门通过需同时满足：

1. **内容审核**：无 critical 问题（warning 需用户确认放行）
2. **图片压缩**：所有图片压缩成功，格式为 WebP
3. **排版格式化（WeChat）**：格式化完成，无 format_warnings 或 warnings 已确认
4. **用户确认**：展示审核/压缩结果后，用户明确确认"通过，继续发布"

全部满足后：
- 将 review 阶段状态设为 `completed`
- 写入 checkpoint（decision: "approved"）
- 进入 publishing 阶段
