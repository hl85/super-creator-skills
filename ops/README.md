## Key Principles

### 1. Single Source of Truth for Versions

**marketplace.json** is the primary version file:

```json
{
  "metadata": {
    "version": "3.0.0"  ← This is the truth
  }
}
```

All other version references are **automatically synchronized** during release via `/release-skills` skill.

### 2. Automated Release Process

- No manual version editing
- Release automation: `ops/scripts/` + `/release-skills` skill + `.releaserc.yml`
- All version files stay in sync automatically
- Release commits and tags created automatically

### 3. Git Flow for Stability

- `main` → Production (stable, tagged)
- `develop` → Integration (testing, feature merging)
- `feature/*` → Individual feature branches
- `release/*` → Temporary release preparation
- `hotfix/*` → Emergency fixes

### 4. Conventional Commits

All commits follow a standard format for automatic changelog generation:

```
feat(skill-name): description      ← New feature
fix(skill-name): description       ← Bug fix
refactor(project): description     ← Refactoring
docs(skill-name): description      ← Documentation
```

## Integration with Tools

### release-skills Skill

Claude Code skill at `.claude/skills/release-skills/`

**Usage**:
```bash
/release-skills                    # Execute release
/release-skills --dry-run          # Preview
/release-skills --patch            # Force patch version
/release-skills --minor            # Force minor version
/release-skills --major            # Force major version
```

**Reads from**:
- `.releaserc.yml` (configuration)
- `ops/release/STRATEGY.md` (documentation)
- git history (commits since last tag)

**Writes to**:
- `marketplace.json` (version)
- `CHANGELOG.md` (English)
- `CHANGELOG.zh.md` (Chinese)
- git commits and tags

### Git Hooks

Installed via `ops/scripts/install-git-hooks.mjs`:

- `pre-push` → Validates version consistency before pushing
- `pre-commit` → (Planned) checks code quality

## FAQ

**Q: Can I manually edit marketplace.json?**

A: No, during release. Between releases, for non-version fields only. Always use `/release-skills` for version changes.

**Q: What if version files get out of sync?**

A: Run `node ops/scripts/verify-version-sync.mjs` to check. If they're out of sync, it indicates the release process wasn't followed correctly.

**Q: Do I need to update all version files for each release?**

A: No. Only `marketplace.json` is edited; others are auto-generated by `/release-skills`.

**Q: Can I skip pre-release checks?**

A: Not recommended. The checklist ensures quality and prevents broken releases.
